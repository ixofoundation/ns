# P2 — Linked-data interoperability: status

Status of the P2 findings from the expert review (2026-06-08). Everything marked
Done is in the working tree and `npm run validate` stays green.

## ✅ P2.6 / P2.7 — External SKOS mappings
- **Countries (P2.7) — Done.** Each `vocab/v1/countries` concept now carries a
  `skos:closeMatch` to a name-derived **DBpedia** resource, and the scheme has
  `dcterms:conformsTo` → ISO 3166-1. `closeMatch` (not `exactMatch`) is used
  deliberately: the alignment is by label, not individually verified — the ISO
  `skos:notation` remains authoritative. Generated by `scripts/build-schemes.mjs`.
- **Other type schemes (P2.6) — Deferred, documented.** The remaining IXO enums
  (oracle-capabilities, node-types, payment-*, stake-types, agent-roles, …) have
  **no clean external authority** to map to (they're IXO-specific governance
  concepts). Fabricating links would be worse than none. When a target vocabulary
  is chosen per scheme, add `skos:exactMatch`/`closeMatch` via the generator
  (the `x.exact` / `x.match` hooks already exist in `build-schemes.mjs`).

## ✅ P2.8 — QUDT / SOSA for measurements (the dMRV core)
Done in `emerging-eco/ns`. The vocabulary now bridges measurement terms:
`emerging:Quantity`/`Carbon`/`EmissionReductionFactor rdfs:subClassOf qudt:Quantity`,
`emerging:Measure rdfs:subClassOf sosa:Observation`, `emerging:amount rdfs:subPropertyOf qudt:value`,
`emerging:units rdfs:subPropertyOf qudt:unit`. A `hasUnit` term (`→ qudt:unit`) was
added to the emerging context, and the `cleanCookingClaim` artefact's `fuelSale`
now typed `qudt:Quantity` with `hasUnit unit:KiloGM`. (`qudt`/`unit`/`sosa`
prefixes were already declared; now actually used.)

## ◐ P2.9 — Time / geo / quantities
- **Quantities — Done** via QUDT (P2.8).
- **Time / geo — Deferred.** Periods still use ad-hoc start/end strings; locations
  use ISO codes / strings. Recommended follow-up: OWL-Time (`time:Interval`) for
  periods and GeoSPARQL / `schema:GeoCoordinates` for locations. Low urgency.

## ✅ P2.10 — Robust `ClaimShape`
Done. The hardcoded `sh:in` list of claim-type IRIs was replaced with a
`skos:inScheme` membership check (`sh:node` → `skos:inScheme sh:hasValue
protocol/claims/v1`), so the constraint **tracks the scheme automatically** with
no drift. `validate-shapes.mjs` merges the claim-types scheme into the data graph.
Verified with a negative test: `claimType:impact` conforms; a non-member
(`entityType:project`) is correctly rejected.

## ◐ P2.11 — Hash vs slash IRIs for large schemes — Decision recorded
`vocab/v1/countries` (250 concepts) uses `#fragment` IRIs, so dereferencing one
country fetches the whole document. **Decision: keep hash IRIs.** Rationale: the
schemes are small-to-moderate single documents, hash namespaces are the common
SKOS convention, and per-concept slash IRIs would require 250+ redirect targets
and per-concept files for marginal benefit. Revisit only if a scheme grows large
enough that whole-document fetches become a real cost.

## ✅ P2.12 — JSON Schema hygiene
- **Drafts aligned — Done.** The three `emerging-eco/ns` schemas were bumped from
  draft 2019-09 to **draft 2020-12**, matching `ixofoundation/ns schema/v1/*`.
  All compile.
- **DID patterns tightened — Done.** `^did:` → `^did:[a-z0-9]+:.+` (and
  `^did:ixo:` → `^did:ixo:.+`) across the IXO schemas.
- **Shared `$defs` — Deferred.** Each schema still carries its own DID/context
  `$defs`. Extracting a shared `schema/v1/_defs.json` referenced via `$ref` is a
  minor DRY improvement (3 files); wire `ajv.addSchema` for all schemas first.
