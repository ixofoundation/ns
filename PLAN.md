## 0. Background

The IXO Spatial Web protocol has multiple JSON-LD-shaped artefacts that don't currently parse to RDF. An audit conducted in May 2026 against the `GraemeLeightonIXO-patch-4` branch surfaced:

- A duplicate `schema:` key in `context/v1/index.jsonld` that silently overwrites schema.org.
- Illegal JSON-LD 1.1 keyword redefinitions (`"@id": "@type"`) across the protocol sub-namespaces.
- A literal `undefined/v1/` folder generated by a buggy build script.
- Missing `index.jsonld` redirects in the `.htaccess`, with several rules pointing at folders or files that don't exist.
- Most JSON-LD files produce zero RDF triples when expanded.

The full audit findings are summarised in the appendix. This plan addresses every audit finding plus the broader work needed to make the namespace functional as the interpretation layer for on-chain entity domains.

## 1. Architectural principles

These principles are settled and should not be relitigated mid-execution.

### 1.1 Separation of concerns

The namespace publishes the **interpretation layer** only. It does not duplicate the on-chain entity graph.

| What lives on Impact Hub | What lives in the namespace |
|---|---|
| Specific entities (`did:ixo:entity:xxx`) | Class definitions (`ixo:Project`, `ixo:Oracle`, …) |
| Controllers, verification methods, accounts | Property definitions (`ixo:linkedClaim`, `ixo:burnTime`, …) |
| `linkedEntity`/`linkedClaim` arrays connecting specific entities | Concept schemes for closed enums (`entityType:`, `status:`, …) |
| Claim and credential instances | Schemas and shapes that validate those instances |

Consumers fetch instances from Impact Hub (via Blocksync) and use the namespace to *interpret* what they're looking at.

### 1.2 Serialization: JSON-LD only

JSON-LD is the canonical and only serialization. No parallel Turtle. The team's downstream pipeline (SDKs, VCs, Matrix, on-chain) is JSON-LD-native, and dual serialization is deferred until external consumers require it.

### 1.3 Canonical URI form

Canonical: `https://w3id.org/ixo/<path>` (no `ns/` prefix).
Variant: `https://w3id.org/ixo/ns/<path>` issues a 301 to the canonical form.

### 1.4 `@protected` policy: selective

Protect W3C-keyword aliases (`id`, `type`) and security-critical terms (proof types, controller, verification methods). Do not blanket-protect every term — that locks downstream consumers out of useful overrides.

### 1.5 Schemas AND shapes

JSON Schema for document-shape validation at API ingress and form generation. SHACL for graph-shape validation after JSON-LD expansion. Both. They validate different things; don't duplicate constraints.

### 1.6 Versioning

All vocabularies live under `v1/`. Breaking changes go to `v2/` with `owl:deprecated` markers on superseded terms and `dcterms:isReplacedBy` links. Non-breaking additions are made in place.

#### Goal

#### Deliverables

- `package.json` with `jsonld`, `ajv`, `ajv-formats`, `shacl-engine` (or `rdf-validate-shacl`) as dependencies.
- `scripts/validate.mjs` — expands (and `toRDF`s) every `.jsonld`/`.json` file that has a top-level `@context`, fails on processor errors. Triples are asserted **≥1 for data documents only**; two categories are exempt: (a) **context-only** documents whose sole top-level key is `@context` (the umbrella and claim contexts legitimately yield 0 triples), and (b) **plain JSON** with no `@context` (impact tokens, JSON Schemas — routed to the schema validator, not JSON-LD expansion). A document loader resolves `https://w3id.org/ixo/*` against the local working copy.
  - **Project rule (amended 2026-06-07): `@id`/`@type` keyword overrides are ALLOWED.** Standard JSON-LD 1.1 rejects `"@id":"@type"` ("keywords cannot be overridden"); the validator now strips *only* those keyword redefinitions from `@context` objects and expands the remainder, flagging the file as *keyword-override tolerated (non-conformant JSON-LD)*. ⚠ External, spec-conformant processors (SDKs, VC libraries, triplestores) still reject such documents and yield zero RDF — this tolerance applies to the in-repo harness only.
- `scripts/validate-schemas.mjs` — validates every `.json` schema file as a valid JSON Schema document.
- `scripts/validate-shapes.mjs` — runs SHACL on every fixture under `fixtures/on-chain/` against shapes in `shapes/v1/`.
- `.github/workflows/validate.yml` — runs all three on every PR.
- `README.md` at repo root describing the layout, canonical URI form, how to run validations locally.
- `scripts/.expect-error.json` allowlist tagging the currently-broken files as `expect-error` so CI doesn't gate the rest of the rebuild. Seeded from the **17 files** carrying the illegal `"@id": "@type"` keyword redefinition (see Appendix A #2) plus any data document that expands to 0 triples (e.g. `activity/v1`, `attributes/index.json`).

- New `README.md` describes the namespace structure and points at `PLAN.md` for the rebuild plan.
